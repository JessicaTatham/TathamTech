I spent years building things at scale. Platforms used by millions of people, systems that couldn't go down, codebases maintained by dozens of developers across multiple teams. The kind of environments where a bad deploy on a Tuesday afternoon ruins everyone's week.

Now I build websites for small businesses. Five pages. Ten pages. A contact form and a blog.

People sometimes ask if that feels like a step down. It doesn't. It feels like bringing a sharper set of tools to a job that deserves better tools.

Let me explain what I mean.

## Performance Isn't an Afterthought

When you build for an enterprise, performance is measured and monitored constantly. Page load times are tracked. Core Web Vitals are reported on. Someone will notice if a deployment makes the site 200 milliseconds slower. You develop an instinct for it. You write efficient code not because you're trying to, but because you've internalized what efficient code looks like.

Most small business websites are built by people who've never operated at that scale. That's fine. You don't need enterprise-scale architecture for a plumber's website. But the habits transfer.

When I build a small site, images are optimized before they hit the page. CSS is minimal and purposeful. JavaScript loads only when it's needed. The hosting is configured for speed. These aren't heroic efforts. They're just how I build things, because working in environments where every millisecond mattered trained me to think that way.

The result is a five-page site that loads in under a second. Not because it needed enterprise engineering, but because the person building it was shaped by enterprise standards.

## Code That Someone Else Can Maintain

In enterprise environments, you never write code just for yourself. You write code that dozens of other developers will read, modify, debug, and extend. Your code goes through peer review. If it's unclear, it gets sent back. If it's clever but unreadable, it gets sent back. If it works but it's fragile, it gets sent back.

That discipline sticks with you.

When I build a small business site, the code is clean. Variables are named clearly. Components are organized logically. There are comments where they matter. If I get hit by a bus (the freelancer's favorite hypothetical), another developer can pick up the project and understand what's happening without spending days reverse-engineering it.

This might seem like a strange thing to care about for a small project. But small business owners switch developers all the time. Relationships change. Needs evolve. The developer who builds your site might not be the one maintaining it in three years. When that handoff happens, the quality of the code determines whether it takes the new developer two hours or two weeks to get up to speed.

I review my own code the way I would review a teammate's. That sounds a little neurotic. It probably is. But the output is better for it.

## Security Is a Habit, Not a Feature

In financial services, security wasn't something you added at the end. It was in every decision, every code review, every architecture discussion. You learned to think about attack vectors the way a locksmith thinks about locks. Not paranoid, just aware.

Small business websites aren't high-value targets in the way a bank is. But they are targets. WordPress sites get hacked constantly. Not because hackers specifically want your plumbing company's customer list, but because automated bots scan the entire internet for vulnerabilities and exploit whatever they find.

I build with security defaults. HTTPS everywhere. Secure headers configured properly. No unnecessary third-party scripts with access to your data. Admin interfaces behind strong authentication. Dependencies updated and monitored. These are basic things, but I'm consistently surprised by how many small business sites get them wrong.

The enterprise made me paranoid in the right ways. Your site benefits from that paranoia even if the stakes are lower.

## Scalability Thinking at Small Scale

This sounds like a contradiction, but hear me out.

When I was working on platforms at Bell, we had to think about what happens when traffic spikes. What happens during a product launch. What happens when the system gets ten times more load than normal. You design for the ceiling, not the floor.

A small business site probably won't get ten times its normal traffic. But it might. You run a promotion that goes viral on social media. You get featured in a local news article. Someone shares your blog post and suddenly five thousand people are on your site instead of fifty.

A well-built site handles that without breaking. Not because it has enterprise infrastructure behind it, but because the architecture is sound. Static generation where possible. Proper caching. A hosting setup that can absorb spikes without falling over or racking up massive charges.

I've built sites for clients that got unexpected traffic surges and handled them without anyone even noticing. That's not luck. That's architecture.

## Process Discipline

Enterprise work teaches you process whether you like it or not. Sprints. Code reviews. Testing. Staging environments. Deployment checklists. Documentation. It can feel like bureaucracy when you're in it, and some of it genuinely is. But the core of it is sound: don't wing it.

For small projects, I don't run two-week sprints or write 40-page technical specifications. That would be absurd. But I do have a process. There's a defined scope. There are milestones. There's a staging environment where you review the site before it goes live. There's a deployment checklist so nothing gets missed at launch. There's documentation for how to use and maintain the site.

This might sound obvious, but a shocking number of freelance web projects have no formal process at all. The developer builds, the client sees it when it's done, and everyone hopes for the best. That's how things get missed, expectations get misaligned, and projects go sideways.

## The Animal Science Detour

Before all of this, I studied animal science. I was going to be a large animal veterinarian. Life had other plans, and I ended up in tech instead. People usually find this amusing or confusing when I mention it.

But I think starting outside of tech gave me something that pure computer science graduates sometimes lack: the ability to explain things to people who aren't technical. When you come to technology from another field, you remember what it felt like to not understand it. You remember the jargon that confused you. You remember the frustration of asking a question and getting an answer that assumed knowledge you didn't have.

I talk to clients in plain language because I remember when all of this was foreign to me too. That's not a pedagogical technique. It's empathy born from experience.

## What You're Actually Getting

When you hire a developer with enterprise experience for your small business project, you're not paying for over-engineering. You're paying for invisible quality. The things you'll never see but will absolutely feel.

Your site will be fast, not because someone spent extra time optimizing it, but because speed was the default. Your code will be maintainable, not because someone documented it for a team, but because clean code is a habit. Your site will be secure, not because someone ran a security audit, but because secure practices were embedded in every decision.

You're paying for the absence of problems. Which, I realize, is a terrible sales pitch. Nobody gets excited about the thing that doesn't break, the vulnerability that doesn't exist, the technical debt that was never created.

But six months after launch, when your site is still fast, still secure, still working exactly the way it should, you'll feel it. You just might not know what to attribute it to.

## Not a Brag. A Perspective.

I'm not arguing that every small business needs to hire someone with a decade of enterprise experience. Plenty of talented developers build great small business sites without ever setting foot inside a Fortune 500 company.

What I'm saying is that the experience transfers in ways that aren't immediately obvious. It's not about the scale of the projects. It's about the discipline they instill. The way you think about a problem changes when you've solved it at scale. The way you write code changes when everything you write gets reviewed. The way you approach security changes when you've built systems that handle sensitive data.

I bring all of that to a five-page website. Not because a five-page website demands it, but because it's who I am as a developer. You can't unlearn these habits, and I wouldn't want to.

The result is a small project built with big-project discipline. Quietly, that makes all the difference.
